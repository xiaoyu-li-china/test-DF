# 陶艺库存管理系统 - 架构设计

## 一、当前 SQLite 表结构

### 1.1 materials 物料表
```sql
CREATE TABLE materials (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    name                TEXT    NOT NULL,                    -- 物料名称
    material_type       TEXT    NOT NULL,                    -- 类型: 泥料/釉料/成品
    quantity            REAL    NOT NULL DEFAULT 0,          -- 当前库存数量
    unit                TEXT    NOT NULL,                    -- 计量单位 (kg/袋/件)
    low_stock_threshold REAL    NOT NULL DEFAULT 0,          -- 低库存预警阈值
    created_at          TEXT    NOT NULL,                    -- 创建时间
    updated_at          TEXT    NOT NULL                     -- 最后更新时间
);
```

### 1.2 transactions 出入库记录表
```sql
CREATE TABLE transactions (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    material_id     INTEGER NOT NULL,                        -- 关联物料ID
    transaction_type TEXT    NOT NULL,                       -- 类型: 入库/出库
    quantity        REAL    NOT NULL,                        -- 变动数量
    operator        TEXT    NOT NULL,                        -- 操作员: 店员1/2/3
    note            TEXT,                                    -- 备注
    created_at      TEXT    NOT NULL,                        -- 操作时间
    FOREIGN KEY (material_id) REFERENCES materials (id)
);
```

### 1.3 索引建议 (当前未创建，建议添加)
```sql
CREATE INDEX idx_materials_type ON materials(material_type);
CREATE INDEX idx_materials_name ON materials(name);
CREATE INDEX idx_transactions_material_id ON transactions(material_id);
CREATE INDEX idx_transactions_created_at ON transactions(created_at);
```

## 二、核心事务边界分析

### 2.1 并发出入库事务 (已修复)

**问题场景**：两个窗口同时出库同一物料，发生更新丢失。

**修复前**（有问题）：
```python
# 非原子操作，两步之间其他进程可能已修改数据
cursor.execute('SELECT quantity FROM materials WHERE id = ?', (mid,))
current_qty = cursor.fetchone()[0]
new_qty = current_qty - qty
cursor.execute('UPDATE materials SET quantity = ? WHERE id = ?', (new_qty, mid))
conn.commit()
```

**修复后**（原子操作）：
```python
conn = get_connection()
conn.execute('BEGIN IMMEDIATE')    # 立即获取写锁
cursor = conn.cursor()

# 原子 UPDATE + 条件检查，数据库级保证一致性
if transaction_type == '入库':
    cursor.execute('''
        UPDATE materials 
        SET quantity = quantity + ?, updated_at = ? 
        WHERE id = ?
    ''', (quantity, now, material_id))
else:
    cursor.execute('''
        UPDATE materials 
        SET quantity = quantity - ?, updated_at = ? 
        WHERE id = ? AND quantity >= ?    -- 库存检查在UPDATE内
    ''', (quantity, now, material_id, quantity))
    
    if cursor.rowcount == 0:    # 受影响行数为0表示库存不足
        conn.rollback()
        raise ValueError('库存不足')

cursor.execute('INSERT INTO transactions ...')
conn.commit()    # 事务提交
conn.close()
```

### 2.2 事务边界与隔离级别

| 操作 | 事务范围 | 隔离级别 | 锁机制 |
|------|----------|----------|--------|
| 查询库存 | 自动提交 | 读已提交 | 共享锁（只读） |
| 入库 | BEGIN IMMEDIATE → UPDATE → INSERT → COMMIT | 读已提交 | 排他写锁（BEGIN立即获取） |
| 出库 | BEGIN IMMEDIATE → UPDATE(带条件) → INSERT → COMMIT | 读已提交 | 排他写锁 + rowcount 检查 |
| 新增/编辑/删除物料 | 自动提交（单条SQL） | 读已提交 | 自动获取 |

### 2.3 关键幂等性保证
- **出库**：`UPDATE ... WHERE quantity >= ?` + `rowcount == 0` 检查，确保不会超卖
- **入库**：`quantity = quantity + ?` 原子递增，重复执行也不会出错
- **所有操作**：通过 `created_at` 记录操作时间，便于审计追踪

## 三、局域网多机同步方案对比

### 3.1 方案选型

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **A. 集中式 SQLite 网络共享** | 改动最小，直接共享db文件 | 容易锁冲突，网络不稳定易损坏 | ❌ 不推荐 |
| **B. 切换到 PostgreSQL/MySQL** | 原生支持多连接，ACID完整 | 需要部署服务器，改动较大 | ⭐⭐⭐ |
| **C. SQLite + 定时同步脚本** | 简单，各机仍可离线用 | 存在同步延迟，冲突难解决 | ⭐⭐ |
| **D. LiteSync / rqlite** | SQLite 原生分布式方案 | 学习成本，配置复杂 | ⭐⭐⭐ |
| **E. 服务端 API 架构** | 清晰解耦，可扩展性强 | 需要重写数据访问层 | ⭐⭐⭐⭐⭐ |

### 3.2 推荐方案：E - 轻量级服务端 API

**架构图**：
```
┌─────────────┐    HTTP JSON    ┌──────────────────┐
│ 桌面客户端1 │ ───────────────> │                  │
├─────────────┤                 │  Flask/FastAPI    │
│ 桌面客户端2 │ ───────────────> │   服务端         │
├─────────────┤                 │                  │
│ 桌面客户端3 │ ───────────────> │  SQLite/PostgreSQL│
└─────────────┘                 └──────────────────┘
```

**改造点**：
1. 服务端：Flask + SQLite，暴露 REST API
2. 客户端：`database.py` 改为 HTTP 调用而非直接访问DB
3. 事务边界移到服务端，由服务端保证并发安全

### 3.3 同步关键技术点

| 需求 | 实现方案 |
|------|----------|
| 实时库存一致 | 所有读写走服务端，服务端用事务保证 |
| 离线可用 | 本地SQLite缓存，在线时双向同步（可选） |
| 操作审计 | 每条 transaction 带客户端标识和时间戳 |
| 冲突解决 | 最后写入胜出（LWW）+ 操作日志回查 |
| 数据备份 | 服务端定时 dump，每日备份 |
