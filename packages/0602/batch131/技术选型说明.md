# 助餐配送路线可视化——技术选型说明

## 一、为什么片区划分选 KMeans 而不是 DBSCAN

咱们项目要按"片区"把几十个送餐点分组，再分派给不同骑手。下面用大白话对比两种常用算法：

### KMeans：提前说好"分几组"

| 特点 | 说明 |
|------|------|
| **需要指定片区数 k** | 比如你有 4 个骑手，就设 k=4，每组对应一个人 |
| **每组必有站点** | 不会出现某个片区空空如也的情况 |
| **边界清晰** | 每个站点只属于一个片区，不会重叠 |
| **结果稳定** | 同一批数据跑出来片区大致一样 |

### DBSCAN：自动发现"聚集区域"

| 特点 | 说明 |
|------|------|
| **不需要指定片区数** | 它自己根据密度决定有几个簇 |
| **可能产生离群点** | 站点稀疏的地方会被标记为"噪声"，没人管 |
| **簇大小不均** | 密集区域可能合并成一大片，稀疏区域可能散落 |
| **参数敏感** | "多远算邻居"和"几个邻居才算簇"这两个值不好调 |

### 为什么 KMeans 更适合咱们

1. **骑手数是固定的**——我们一般就 3-5 个骑手，KMeans 的 `k=骑手数` 直接对应，DBSCAN 不保证分出来的簇数和骑手数匹配。

2. **不能有老人被漏掉**——DBSCAN 会把偏远的站点标成"噪声"丢出去，但助餐项目每个老人都得送到。KMeans 保证每个点都有归属。

3. **每个骑手工作量要均衡**——KMeans 分出来的片区大小比较均匀，不会出现一个骑手跑 20 站、另一个只跑 3 站的情况。

4. **结果可复现**——给街道领导汇报时，每次跑出来片区划分一样，不会"今天海淀分两片、明天分三片"。

> **一句话总结**：KMeans 是"按编制分班"，DBSCAN 是"自然聚堆"。助餐配送有固定的骑手编制和"人人必须覆盖"的要求，所以"按编制分班"更合适。

## 二、地图上的折线顺序和 TSP 是什么关系

### 什么是 TSP

TSP（旅行商问题）就是经典的"走遍所有点再回到起点，怎么走最短"的问题。

在我们的场景里：中央厨房 → 各个送餐点 → 回到中央厨房，找一条总路程最短的路线。

### 为什么不直接求最优解

TSP 是 NP-hard 问题——站点数一多（比如 20 个点），所有可能路线有 `19! ≈ 1.2×10¹⁷` 条，超级计算机也算不完。所以实际都用**近似算法**。

### 我们用的最近邻贪心算法

代码里的 `nearest_neighbor_route()` 就是最近邻贪心法，逻辑很简单：

```
1. 从厨房出发
2. 看哪个还没去的站点离当前最近 → 去那儿
3. 重复步骤 2，直到所有站点都去过
4. 回到厨房
```

这就是地图上**彩色折线的连接顺序**——折线从厨房画到第 1 站，再到第 2 站，依此类推。折线上的数字标注就是配送顺序号。

### 贪心法的优缺点

| 优点 | 缺点 |
|------|------|
| 速度极快，55 个点瞬间算完 | 不保证全局最短，可能绕路 |
| 逻辑简单，同事都能看懂 | 可能出现"交叉路线" |
| 实际效果够用，一般比最优解长 20%-30% | 站点越少越接近最优 |

### 对实际配送的影响

- **20%-30% 的多绕**听起来多，但城区路网本身就不能直线走，这个差距在实际配送中感知不大
- 如果对路线质量有更高要求，可以换成 **2-opt 优化**（把交叉的线段"解扣"），效果更好但代码也更复杂
- 目前这个方案在**15-20 站/骑手**的规模下完全够用

> **一句话总结**：地图上折线的连接顺序就是贪心 TSP 近似解，不是数学上的最短路线，但足够实用。想要更优可以换 2-opt，目前没那个必要。
